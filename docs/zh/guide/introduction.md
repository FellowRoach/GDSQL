# 简介

**GDSQL** 是一个基于 Godot 引擎 `ConfigFile` 系统构建的数据库 SQL 工作台插件。它为需要轻量级、版本控制友好的数据库的游戏开发者提供了完整的数据管理方案，数据库文件直接存放在项目目录中。

## GDSQL 是什么？

GDSQL 将所有数据存储在 `.cfg` 纯文本文件中——这是 Godot 原生支持的配置格式。不需要外部数据库引擎、不需要服务器进程、不需要额外的运行时依赖。安装插件即可使用。

### 核心能力

| 功能 | 说明 |
|------|------|
| SQL 查询引擎 | 完整 SQL 语法：SELECT、INSERT、UPDATE、DELETE、REPLACE，支持 JOIN、子查询、聚合函数 |
| 可视化工作台 | Excel 风格数据编辑、树形浏览器、表结构编辑器、差异视图 |
| 链式 DAO API | 纯 GDScript 方法链实现所有 CRUD 操作 |
| GBatis ORM | MyBatis 风格 XML 映射，支持动态 SQL 和结果映射 |
| 映射图编辑器 | 可视化拖拽编辑器，设计表关联并生成代码 |
| 数据加密 | AES-256-CBC 加密，DEK 密钥体系和密码验证 |
| XML 编辑器 | 内置编辑器，支持语法高亮和多标签页 |
| 多语言 | 内置 13 种语言支持 |

## 为什么选择 GDSQL？

### 对比 SQLite / GDSQLite

SQLite 将数据存储在二进制文件中——不适合版本控制 diff。GDSQL 使用 `.cfg` 纯文本，每次更改都在 `git diff` 中可见。GDSQL 还提供了 SQLite 所没有的可视化编辑器和 ORM 框架。

### 对比自定义 Resource 文件

Godot 的 `.tres`/`.res` 文件可以 diff，但没有查询能力。GDSQL 在文本格式之上提供了完整的 SQL 引擎、Excel 风格编辑和 ORM 映射。

## GDSQL 适合谁？

- **游戏开发者**——管理大量配置数据（道具、技能、关卡、NPC 等）
- **团队**——需要从设计到运行时的统一数据工作流
- **开发者**——偏好 SQL 风格的查询而非自定义数据访问代码
- **项目**——需要版本控制友好的数据文件

## 局限性概览

- 纯 GDScript 实现——不适用于高吞吐量实时场景
- 针对数百到数万条记录优化
- 不支持并发写入（仅主线程）
- 无服务器模式——仅本地文件存储
- 不支持 DDL SQL 语句（通过可视化界面修改结构）

详见 [局限性](./limitations) 页面。
